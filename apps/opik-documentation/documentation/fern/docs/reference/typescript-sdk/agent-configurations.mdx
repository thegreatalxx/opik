The Opik TypeScript SDK provides agent configuration management using [Zod](https://zod.dev) schemas for type-safe, versioned configuration objects. You can publish configuration versions, deploy them to environments, and fetch them at runtime with automatic fallback behavior.

## Overview

Agent configurations are defined as Zod object schemas and managed through two main client methods:

- **`createAgentConfig`** — Publishes a new configuration version (or reuses an existing one if values match)
- **`getAgentConfigVersion`** — Fetches a configuration version at runtime inside a `track()` context

The returned `AgentConfig<T>` object is a typed, read-only object with additional metadata properties.

## Defining a Schema

Define your configuration shape as a Zod object schema. The schema **must** have a `.describe()` name — this is used as the configuration's identifier:

```typescript
import { z } from "zod";
import { Prompt, ChatPrompt } from "opik";

const MyConfigSchema = z.object({
    temperature: z.number().describe("Sampling temperature"),
    model: z.string(),
    maxTokens: z.number(),
    systemPrompt: z.instanceof(ChatPrompt).describe("System prompt for the agent"),
    useCot: z.boolean().describe("Enable chain-of-thought reasoning"),
}).describe("MyConfig");
```

Use `.describe()` on individual fields to attach human-readable descriptions. Use `z.instanceof(Prompt)` or `z.instanceof(ChatPrompt)` for prompt fields.

## `createAgentConfig`

Publishes a typed agent config version to Opik. If an identical version already exists, it is reused; otherwise a new version is created.

### Signature

```typescript
async createAgentConfig<S extends z.ZodObject<z.ZodRawShape>>(
    schema: S,
    values: z.infer<S>,
    options?: {
        projectName?: string;
        description?: string;
    }
): Promise<string>
```

### Parameters

| Parameter | Type | Required | Description |
|---|---|---|---|
| `schema` | `z.ZodObject` | Yes | Zod object schema with a `.describe()` name that defines the config shape. |
| `values` | `z.infer<S>` | Yes | Typed config values that conform to the schema. |
| `options.projectName` | `string` | No | Project to publish under. Defaults to the client's configured project. |
| `options.description` | `string` | No | Human-readable description for this version. |

### Returns

`Promise<string>` — The version name (or ID) of the published config.

### Example

```typescript
import { Opik, ChatPrompt } from "opik";
import { z } from "zod";

const client = new Opik();

const systemPrompt = await client.getChatPrompt({ name: "my-system-prompt" });

const MyConfigSchema = z.object({
    temperature: z.number().describe("Sampling temperature"),
    model: z.string(),
    maxTokens: z.number(),
    systemPrompt: z.instanceof(ChatPrompt).describe("System prompt for the agent"),
}).describe("MyConfig");

const version = await client.createAgentConfig(
    MyConfigSchema,
    { temperature: 0.7, model: "gpt-4", maxTokens: 1024, systemPrompt },
    { projectName: "my-project", description: "Production v1" },
);

console.log(`Published version: ${version}`);
```

## `getAgentConfigVersion`

Retrieves a typed agent config version and returns it as an `AgentConfig<T>` object. **Must be called inside a `track()` function** — it automatically attaches the resolved config metadata to the active trace.

### Signature

```typescript
async getAgentConfigVersion<S extends z.ZodObject<z.ZodRawShape>>(
    schema: S,
    options: {
        fallback: z.infer<S>;
        projectName?: string;
        env?: string;
        latest?: boolean;
        version?: string;
    }
): Promise<AgentConfig<z.infer<S>>>
```

### Parameters

| Parameter | Type | Required | Description |
|---|---|---|---|
| `schema` | `z.ZodObject` | Yes | Zod object schema with a `.describe()` name. |
| `options.fallback` | `z.infer<S>` | Yes | Values to use when the backend is unreachable. |
| `options.projectName` | `string` | No | Project to fetch from. Defaults to the client's configured project. |
| `options.env` | `string` | No | Environment label to resolve. Defaults to `"prod"` when no other selector is provided. |
| `options.latest` | `boolean` | No | If `true`, fetch the most recently published version regardless of env. |
| `options.version` | `string` | No | Fetch a specific named version. |

### Version Selectors

Exactly one of the following selectors can be provided (they are mutually exclusive):

| Selector | Description |
|---|---|
| `env` | Fetches the version pinned to that environment label (default: `"prod"`) |
| `latest: true` | Fetches the most recently published version |
| `version` | Fetches the named version exactly |

If none is provided, `env: "prod"` is used by default.

### Returns

`Promise<AgentConfig<z.infer<S>>>` — A typed, read-only config object with the resolved values and metadata.

### Example

```typescript
import { Opik, track, ChatPrompt } from "opik";
import { z } from "zod";

const client = new Opik();

const fallbackPrompt = await client.getChatPrompt({ name: "my-system-prompt" });

const MyConfigSchema = z.object({
    temperature: z.number(),
    model: z.string(),
    maxTokens: z.number(),
    systemPrompt: z.instanceof(ChatPrompt),
}).describe("MyConfig");

const result = await track({ name: "my-agent" }, async () => {
    const config = await client.getAgentConfigVersion(MyConfigSchema, {
        fallback: { temperature: 0.5, model: "gpt-4o-mini", maxTokens: 512, systemPrompt: fallbackPrompt },
        projectName: "my-project",
        env: "prod",
    });

    console.log(`Model: ${config.model}, Temp: ${config.temperature}, Prompt: ${config.systemPrompt}`);
    return callLLM(config.model, config.temperature);
});

await client.flush();
```

## `AgentConfig<T>` Type

The object returned by `getAgentConfigVersion` is `AgentConfig<T>`, which extends your config values with metadata properties:

```typescript
type AgentConfig<T> = Readonly<T> & AgentConfigMeta;
```

### Properties

| Property | Type | Description |
|---|---|---|
| `blueprintId` | `string \| undefined` | ID of the resolved blueprint. |
| `blueprintVersion` | `string \| undefined` | Version name of the resolved blueprint. |
| `envs` | `string[] \| undefined` | Environment tags associated with the resolved version. |
| `isFallback` | `boolean` | `true` if local fallback values are being used because the backend was unreachable. |

### Methods

| Method | Signature | Description |
|---|---|---|
| `deployTo` | `(env: string) => Promise<void>` | Tags the current version with an environment name (e.g., `"prod"`, `"staging"`). |

### Fallback Behavior

When the backend is unreachable, `getAgentConfigVersion` returns the `fallback` values wrapped as an `AgentConfig<T>` with `isFallback: true`:

```typescript
const result = await track({ name: "my-agent" }, async () => {
    const config = await client.getAgentConfigVersion(MyConfigSchema, {
        fallback: { temperature: 0.0, model: "gpt-3.5-turbo", maxTokens: 512, systemPrompt: fallbackPrompt },
        env: "prod",
    });

    if (config.isFallback) {
        console.log("Using fallback config — backend was unreachable");
    }

    return process(config);
});
```

## Prompts in Configurations

Agent configurations have a direct connection to Opik's prompt management system. Use `z.instanceof(Prompt)` or `z.instanceof(ChatPrompt)` to include prompt fields in your schema. **Updating a prompt will automatically update the configuration** — when a prompt referenced by a configuration is updated, the configuration will reflect the updated prompt the next time it is fetched, without needing to publish a new configuration version.

```typescript
import { Opik, Prompt, ChatPrompt } from "opik";
import { z } from "zod";

const client = new Opik();

const systemPrompt = await client.getChatPrompt({ name: "my-system-prompt" });
const summaryPrompt = await client.getPrompt({ name: "my-summary-prompt" });

const AgentWithPromptsSchema = z.object({
    systemPrompt: z.instanceof(ChatPrompt).describe("System prompt for the agent"),
    summaryPrompt: z.instanceof(Prompt).describe("Summary prompt template"),
    temperature: z.number(),
}).describe("AgentWithPrompts");

await client.createAgentConfig(
    AgentWithPromptsSchema,
    { systemPrompt, summaryPrompt: summaryPrompt!, temperature: 0.7 },
    { projectName: "my-project" },
);
```

## Trace Metadata

When `getAgentConfigVersion` is called inside a `track()` function, the configuration metadata (blueprint ID, version, field values) is automatically injected into the current trace and span metadata under the `agent_configuration` key.

## Advanced: Configuration Masks

<Note>
  Masks are an advanced feature primarily used by **Opik optimizers** to run experiments with different parameter values. Most users do not need to interact with masks directly — they are managed automatically when you use Opik's optimization and evaluation tools.
</Note>

A **mask** is a partial override that sits on top of an existing configuration. It lets you substitute a subset of fields without creating a full new version. When a mask is active, `getAgentConfigVersion` transparently merges the mask's values over the base configuration, so your agent code doesn't need to change.

### How Masks Work

1. **Create a mask** using `AgentConfigManager.createMask()` — specify only the fields you want to override
2. **Activate the mask context** using `agentConfigContext()` — wrap your agent call so the SDK knows which mask to apply
3. **Fetch the config as usual** — `getAgentConfigVersion` automatically overlays the mask

```typescript
import { Opik, track } from "opik";
import { AgentConfigManager, agentConfigContext } from "opik";
import { z } from "zod";

const client = new Opik();

const MyConfigSchema = z.object({
    temperature: z.number(),
    model: z.string(),
    maxTokens: z.number(),
}).describe("MyConfig");

// Create a mask with overridden values
const manager = new AgentConfigManager("my-project", client);
const maskId = await manager.createMask({
    values: [{ key: "MyConfig.temperature", value: 0.9, type: "float" }],
});

// Run the agent inside the mask context
const result = await agentConfigContext(maskId, async () => {
    return track({ name: "run-agent" }, async () => {
        // getAgentConfigVersion automatically applies the mask
        const config = await client.getAgentConfigVersion(MyConfigSchema, {
            fallback: { temperature: 0.0, model: "gpt-3.5-turbo", maxTokens: 512 },
            latest: true,
        });
        return `Temperature: ${config.temperature}`; // 0.9 from the mask
    });
});
```

### When Are Masks Used?

- **Optimization runs** — Opik optimizers automatically create masks to evaluate different parameter combinations (e.g., testing `temperature=0.5` vs `temperature=0.9`) without modifying the base configuration.
- **A/B testing** — You can manually create masks to run controlled experiments comparing different configuration variants.

The mask ID is recorded in trace metadata (under `_mask_id`), so you can correlate experimental results with the exact parameter overrides that were applied.

## Complete Example

```typescript
import { Opik, track, ChatPrompt } from "opik";
import { z } from "zod";

const client = new Opik();

// 1. Define the configuration schema
const MyConfigSchema = z.object({
    temperature: z.number().describe("Sampling temperature"),
    model: z.string(),
    maxTokens: z.number(),
    systemPrompt: z.instanceof(ChatPrompt).describe("System prompt for the agent"),
}).describe("MyConfig");

// 2. Publish a version
const systemPrompt = await client.getChatPrompt({ name: "my-system-prompt" });
const version = await client.createAgentConfig(
    MyConfigSchema,
    { temperature: 0.7, model: "gpt-4", maxTokens: 1024, systemPrompt },
    { projectName: "my-project", description: "Production v1" },
);
console.log(`Published: ${version}`);

// 3. Fetch at runtime and deploy to an environment
const result = await track({ name: "run-agent" }, async () => {
    const cfg = await client.getAgentConfigVersion(MyConfigSchema, {
        fallback: { temperature: 0.0, model: "gpt-3.5-turbo", maxTokens: 512, systemPrompt },
        projectName: "my-project",
        latest: true,
    });

    // Deploy to prod
    await cfg.deployTo("prod");

    return `Using ${cfg.model} with temp=${cfg.temperature}, prompt=${cfg.systemPrompt}`;
});

console.log(result);
await client.flush();
```
