Agent Configuration is Opik's system for versioning and managing everything that influences your agent's behavior — prompts, model parameters, tool definitions, and more — outside of your codebase.

## Why use Agent Configuration

When building production agents, the parameters that matter most — prompts, temperatures, tool descriptions — are often buried in source code. Agent Configuration lets you manage them as versioned, deployable artifacts so you can iterate without redeploying.

## Key features

{/* TODO: Review and expand each bullet with any additional capabilities */}

- **Version control**: Every configuration change creates an immutable version with a full audit trail
- **Environment labels**: Promote a version to `prod` or `staging` through the UI without touching code
- **Prompt management**: Track and version text prompts and multi-turn chat prompts with variable substitution
- **Typed parameters**: Store model names, temperatures, tool definitions, and any numeric or string values alongside your prompts
- **SDK support**: First-class Python and TypeScript SDK support for defining and fetching configurations

## How it works

{/* TODO: Consider adding a diagram here (reference /img/v2/agent-configuration/ for existing assets) */}

At a high level, Agent Configuration works in two steps:

1. **Define your configuration** — Use the SDK to declare a `Config` subclass (Python) or a plain object (TypeScript) with your fields and default values, then push the first version with `create_config` / `createConfig`.
2. **Fetch at runtime** — Your agent calls `get_or_create_config` (Python) or `getOrCreateConfig` (TypeScript) with an `env` label or a specific `version`. Opik returns the matching configuration, or the provided fallback if the backend is unreachable.

When you want to update your agent's behavior, create a new version in the Opik UI or SDK and move the `prod` label to it. Your running agent picks up the change on its next fetch — no redeployment needed.
