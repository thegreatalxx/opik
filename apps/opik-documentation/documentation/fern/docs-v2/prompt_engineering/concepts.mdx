When building agents, there are many parameters that influence behavior — prompts, model choices, temperature settings, tool definitions, and more. Agent Configuration lets you version and manage all of these in one place, and update them in production without redeploying your code.

## Agent Configurations

An Agent Configuration is a versioned bundle of all the parameters that define your agent's behavior. This includes:

- **Prompts**: System prompts, user prompt templates, and multi-turn chat prompt templates
- **Model parameters**: Model name, temperature, top-p, and other LLM settings
- **Tool definitions**: Descriptions and schemas for tools your agent can call
- **Custom parameters**: Any other string or numeric values relevant to your agent (e.g., RAG retrieval thresholds)

By storing these together, you get a single source of truth for how your agent is configured at any point in time.

## Versions

Every change to an Agent Configuration creates a new immutable version (`v1`, `v2`, `v3`, etc.). Once a version is created, it cannot be modified. This gives you a complete audit trail of how your agent's configuration has evolved and makes it easy to roll back if needed.

You can create new versions through the SDK or the Opik UI. See the [Getting started](/prompt_engineering/getting-started) guide for details.

## Environments

Environments are labels you assign to a specific version to control which configuration your agent fetches at runtime. You can manage environment labels through the Opik UI without touching code.

Supported environment labels:
- **`prod`**: The production version of the configuration (default when no `env` or `version` is specified)
- **Custom labels**: Any label you create in the Opik UI (e.g., `staging`, `canary`)

This means you can update your agent's behavior in production by simply moving the `prod` label to a new version in the UI — no redeployment required.

### Fetching by version

Instead of using environment labels, you can also fetch a specific version directly using the `version` parameter:

- **`latest`**: The most recently created version
- **`v3`** (or any version name): A specific pinned version

<Note>
  You can specify `env` **or** `version`, but not both. If neither is specified, the SDK defaults to `env="prod"`.
</Note>

## Fallback configuration

When calling `get_or_create_config` / `getOrCreateConfig`, you can provide a `fallback` — a set of hardcoded default values that the SDK uses as a safety net.

On your first deploy, if no configuration exists yet in the project, the SDK automatically creates one from your fallback values and returns it. Once that initial version is stored, subsequent calls fetch it from the backend. The SDK caches the result in memory (default TTL: 5 minutes) and refreshes it in the background, so your agent never blocks on a network call in steady state. If Opik becomes temporarily unreachable, the SDK serves the cached value — or, as a last resort, your hardcoded fallback.

You can check whether the returned configuration came from the fallback by inspecting `is_fallback` (Python) or `isFallback` (TypeScript). This is useful for logging or alerting when your agent is running on local defaults rather than a backend-managed version.

If no `fallback` is provided and the backend is unreachable or no configuration exists, the SDK raises a `ConfigNotFound` error. For production agents, we recommend always providing a fallback.

See [Guaranteed availability](/prompt_engineering/advanced/guaranteed-availability) for details on caching and fallback behavior.

## Configuration types

Each field in an Agent Configuration has a type that determines how it is stored and versioned:

| Type | Description | Example use case |
|------|-------------|-----------------|
| Prompt | Text prompts with variable substitution | System prompts, user prompt templates |
| Chat Prompt | Lists of chat messages | Multi-turn prompt templates |
| String | Plain string values | Model names, tool definitions |
| Float | Numeric values | Temperature, top-p, retrieval thresholds |
